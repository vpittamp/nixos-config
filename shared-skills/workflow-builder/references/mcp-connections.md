# MCP Connections

Scope: how workflow-builder agents and `durable/run` workflow steps receive MCP tools, especially project-level ActivePieces piece MCP servers with OAuth-backed credentials.

## Mental model

MCP tools reach an agent through resolved server config:

```
agent config / workflow durable/run config
  mcpConnectionMode + mcpServers
      |
      v
workflow-builder BFF or workflow-orchestrator resolver
  reads mcp_connection rows for the workflow project
  resolves app_connection.external_id when credentials are needed
      |
      v
AgentRuntime.spec.mcpServers or child workflow input
      |
      v
agent-runtime-controller
  writes DAPR_AGENT_PY_BOOTSTRAP_MCP_SERVERS_JSON
      |
      v
dapr-agent-py bootstraps MCP clients and registers tools
```

For ActivePieces piece MCP servers, the service backing is generated by the `activepieces-mcps` GitOps app in stacks. It reads enabled `mcp_connection` rows and pinned pieces, creates cluster-local Knative KServices (`ap-<piece>-service`), and writes the `activepieces-mcp-catalog` ConfigMap used by the UI for search/predefined choices.

## Modes

`AgentConfig.mcpConnectionMode` controls how project-level MCP rows are used:

| Mode | Meaning |
| --- | --- |
| `auto` | Default. Direct/published agent resolution includes project connections when there are no explicit servers; workflow execution resolves project MCP connections. |
| `project` | Include enabled project `mcp_connection` rows in addition to matching requested/profile servers. Use this when the agent should use the workspace's configured tools. |
| `explicit` | Use only explicit `mcpServers` entries. Use this for tightly pinned tool surfaces or tests. |

Workflow execution also supports unresolved server references in `mcpServers`; the resolver matches by `name`, `serverName`, `server_name`, `pieceName`, `serverKey`, or `displayName`.

## ActivePieces piece MCP

An OAuth-backed piece connection has two DB layers:

- `app_connection.external_id`: encrypted credential envelope for the user's connected app.
- `mcp_connection.connection_external_id`: project-level MCP row pointing at that credential.

Resolved runtime config looks like:

```json
{
  "name": "piece_microsoft-outlook",
  "server_name": "piece_microsoft-outlook",
  "sourceType": "nimble_piece",
  "pieceName": "microsoft-outlook",
  "transport": "streamable_http",
  "url": "http://ap-microsoft-outlook-service.workflow-builder.svc.cluster.local/mcp",
  "headers": {
    "X-Connection-External-Id": "conn_..."
  }
}
```

The header is the auth boundary. `piece-mcp-server` uses it to call workflow-builder's internal decrypt API. Do not put OAuth tokens, decrypted credential JSON, or user secrets into workflow specs, agent markdown, KService env, or GitOps manifests.

The caller-facing URL should not include `:3100` for generated ActivePieces KServices. The container listens on 3100 behind Knative, but callers use the KService URL with `/mcp`.

## Authoring patterns

### Use all project MCP connections

Prefer configuring this on the agent in the UI/agent markdown rather than every workflow:

```yaml
mcp_connection_mode: project
```

Then a workflow can simply reference the agent:

```json
{
  "call": "durable/run",
  "with": {
    "agentRef": { "id": "agt_..." },
    "prompt": "Find my latest Outlook email from Contoso and summarize it.",
    "mode": "execute_direct"
  }
}
```

### Request a specific project MCP connection

Use an unresolved identity and let the resolver fill URL/auth from `mcp_connection`:

```json
{
  "call": "durable/run",
  "with": {
    "agentRef": { "id": "agt_..." },
    "prompt": "List the Outlook tools you can access.",
    "mode": "execute_direct",
    "mcpServers": [
      { "pieceName": "microsoft-outlook" }
    ]
  }
}
```

### Direct custom MCP endpoint

Use explicit config only for servers that are not project-catalog managed:

```json
{
  "name": "my-server",
  "transport": "streamable_http",
  "url": "https://example.internal/mcp",
  "headers": { "Authorization": "Bearer ${TOKEN_FROM_SAFE_SOURCE}" }
}
```

Do not inline real secrets in committed workflow specs. For project/user credentials, create an app connection and bind an MCP connection instead.

## Debug checklist

```bash
ctx=dev
slug=test1

kubectl --context "$ctx" -n workflow-builder get cm activepieces-mcp-catalog \
  -o jsonpath='{.data.servers\.json}' | jq .

kubectl --context "$ctx" -n workflow-builder get ksvc \
  -l app.kubernetes.io/managed-by=activepieces-mcp-reconciler \
  -o custom-columns=NAME:.metadata.name,READY:.status.conditions[?(@.type==\"Ready\")].status,URL:.status.address.url

kubectl --context "$ctx" -n workflow-builder get deploy "agent-runtime-${slug}" -o json | \
  jq -r '.spec.template.spec.containers[] | select(.name=="dapr-agent-py") | .env[] | select(.name=="DAPR_AGENT_PY_BOOTSTRAP_MCP_SERVERS_JSON").value' | jq .

kubectl --context "$ctx" -n workflow-builder logs deploy/"agent-runtime-${slug}" \
  -c dapr-agent-py --tail=250 | rg 'mcp-bootstrap|Loaded|Registered|Missing credentials'
```

Success looks like `[mcp-bootstrap] connected ...` plus tool registration. `Missing credentials` usually means the resolved config lacks `X-Connection-External-Id` or the referenced app connection cannot decrypt.

## When to use gitops

Use the `gitops` skill when the issue is platform deployment rather than workflow authoring: missing `activepieces-mcps`, unhealthy Knative KServices, stale `piece-mcp-server` image, `allow-zero-initial-scale`, ArgoCD app health, or promotion to dev/staging/ryzen.
